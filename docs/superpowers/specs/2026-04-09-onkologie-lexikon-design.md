# Oncology Knowledge Lexicon — Design Spec

**Date:** 2026-04-09
**Branch:** `feat/llm-synthesis-expert-knowledge`
**Workstream:** 2 of 2 (companion: `2026-04-09-viz-coverage-design.md`)
**Status:** Approved for implementation planning

---

## 1. Problem Statement

Today, the promptclub MCP server returns raw API fields from
clinicaltrials.gov, openFDA, OpenTargets, PubMed, and Vertex Google Search
without any clinical context enrichment. A trial in `Phase 3` is just the
string "Phase 3". An endpoint of "Overall Survival" is just text. There is no
expert knowledge in the data layer that could help downstream consumers — the
LLM, the visualization recipes, or future analytics — read the data the way a
clinical oncologist would.

The user wants **expert knowledge injected into the system** so that synthesis
quality improves without depending on the consumer LLM having strong oncology
training. The chosen approach (after brainstorming) is a **static curated
lexicon**, not a RAG system or a prompt-engineering scaffold. Initial scope is
**oncology**, with ~250–400 entries, generated by an LLM with **mandatory
manual review**.

This is a deliberate, contained first step. It is independent of and runnable
in parallel with the Visualization Coverage workstream.

## 2. Goals & Non-Goals

### Goals

- A **static, curated oncology lexicon** of ~250–400 entries living in the
  repo as YAML.
- A **deterministic enrichment layer** that walks Pydantic tool responses,
  matches lexicon terms against string fields, and attaches knowledge
  annotations to the response.
- A **curation pipeline** (offline scripts) that uses an LLM to draft entries
  from a manually maintained seed-topic list and produces a review worksheet.
- **Hard quality gates** at the schema level (mandatory sources, source-domain
  allowlist, alias uniqueness) that prevent unverified hallucinations from
  reaching production.
- **Recipe integration** so visualizations can opt in to displaying
  annotations as tooltips, footnotes, or sublabels.
- A **review lifecycle** (`llm-generated` → `reviewed` → `expert-approved`)
  that makes verification status visible without blocking initial delivery.

### Non-Goals

- No RAG / vector index / dynamic retrieval. The lexicon is in-memory and
  static.
- No LLM prompt engineering. The synthesis prompt in
  [app/main.py](../../../app/main.py) is unchanged. Knowledge enters through
  the data layer, not the prompt.
- No other therapeutic areas in v1 (the schema is generic enough for later
  expansion).
- No automatic source-URL verification (manual review is the defense).
- No re-generation loop or staleness detector in v1 (`last_reviewed` date is
  visible, future v2 may add a re-review workflow).
- No crowd-source correction UI.
- No translation (lexicon is English).

## 3. Architecture

```
┌──────────────────────────────────────────────────────┐
│  Knowledge Asset (static, curated)                   │
│  app/knowledge/oncology/                             │
│    ├── lexicon.yaml          (~300 entries)          │
│    ├── schema.py             (Pydantic schema)       │
│    └── loader.py             (loader + index)        │
└──────────────────────────────────────────────────────┘
                       │
                       │ consumed by
                       ▼
┌──────────────────────────────────────────────────────┐
│  Enrichment Layer                                    │
│  app/services/enrichment.py                          │
│                                                      │
│  enrich(response: BaseToolResponse) -> Response      │
│    walks the response tree, finds matching terms,    │
│    attaches .knowledge_annotations field per record  │
└──────────────────────────────────────────────────────┘
                       │
                       ▼
        Orchestrator calls enrich() before returning
                       │
                       ▼
        Pydantic models now carry knowledge_annotations
                       │
                       ▼
        Recipes may render these fields visually
                       │
                       ▼
        LLM receives enriched data in artifact + tool text
```

### Two parallel sub-pipelines

```
A) RUNTIME PIPELINE (server, every query)
   loader → enrichment → orchestration → envelope

B) CURATION PIPELINE (offline, separate)
   seed_topics.yaml → llm_generator.py → review_worksheet.csv
                                       → manual review
                                       → lexicon.yaml (committed)
```

The two share only the **schema** (`app/knowledge/oncology/schema.py`) and the
final `lexicon.yaml`. Curation is an offline workflow that does not exist in
the running server.

### Core design principles

1. **Static & deterministic** — YAML in repo, in-memory at runtime, no external service.
2. **Enrichment, not mutation** — original API responses remain visible alongside annotations.
3. **Opt-in per recipe** — recipes decide whether to render annotations.
4. **YAML is source of truth** — curators edit YAML, not Python.
5. **LLM-generated + manually reviewed** — with hard schema gates.

## 4. Schema

`app/knowledge/oncology/schema.py`:

```python
class LexiconEntry(BaseModel):
    id: str                    # stable slug, e.g. "endpoint-os"
    term: str                  # canonical term, e.g. "Overall Survival"
    aliases: list[str]         # ["OS", "overall-survival"]
    category: Literal[
        "trial-phase",
        "trial-status",
        "endpoint",
        "study-design",
        "tumor-type",
        "biomarker",
        "drug-class",
        "response-criterion",   # RECIST, iRECIST, etc.
        "treatment-line",       # 1L, 2L, maintenance, etc.
        "resistance-mechanism",
    ]
    short_definition: str      # 1-2 sentences, plain language
    clinical_context: str      # 2-4 sentences, what this means in trial reading
    typical_values: dict[str, str] | None
    related_terms: list[str]   # ids of related entries
    sources: list[Source]      # min. 1 — required
    review_status: Literal["llm-generated", "reviewed", "expert-approved"]
    last_reviewed: date | None

class Source(BaseModel):
    kind: Literal["nci-thesaurus", "fda-label", "publication", "guideline", "definition"]
    url: str | None
    citation: str
```

Schema enforces:

- `sources` ≥ 1 entry (Pydantic field validator). No sourceless hallucinations.
- `category` is enum-bounded. No category sprawl.
- `review_status` lifecycle is explicit and visible.

## 6. Curation Pipeline

This is the most safety-critical section. LLM-generated clinical content
without hard gates is dangerous.

### Step 1 — Seed topics

`app/knowledge/oncology/seed_topics.yaml` is hand-maintained, grouped by
category:

```yaml
trial-phase:
  - "Phase 1"
  - "Phase 1/2"
  - "Phase 2"
  - "Phase 2/3"
  - "Phase 3"
  - "Phase 4 / Post-Marketing"

endpoint:
  - "Overall Survival"
  - "Progression-Free Survival"
  - "Objective Response Rate"
  - "Duration of Response"
  - "Disease Control Rate"
  - "Time to Progression"
  - "Event-Free Survival"
  # ...

response-criterion:
  - "RECIST 1.1"
  - "iRECIST"
  - "Lugano (lymphoma)"
  - "Cheson (leukemia)"
  # ...

# ~300 terms total across categories
```

The seed file is the **only** place where topic scope is decided. Nothing is
generated that is not seeded. Curation reach is bounded by the seed list.

### Step 2 — LLM generation

`scripts/curation/generate_lexicon.py`:

1. Read `seed_topics.yaml`
2. For each term:
   1. Build prompt with system role ("clinical oncology terminology curator"),
      strict instructions, the JSON schema (exported from Pydantic), and 2–3
      few-shot anchor examples
   2. Call Claude (anthropic SDK) with structured output mode
   3. Validate against `LexiconEntry` schema
   4. On validation failure: retry once with error feedback, then skip + log
3. Write all generated entries to `scripts/curation/output/draft_<date>.yaml`
4. Generate `review_worksheet.csv` with columns:
   `id | term | category | short_definition | clinical_context | source_count | sources_summary | review_status | reviewer_notes | accept | edit | reject`

**Hard requirements baked into the prompt:**

- "Every entry MUST cite at least one source. If you cannot cite, return null and we will skip this term."
- "Source URLs must come from authoritative domains: ncit.nci.nih.gov, fda.gov, ema.europa.eu, recist.eortc.org, pubmed.ncbi.nlm.nih.gov, ctep.cancer.gov."
- "Do not include speculative information. If clinical_context is uncertain, leave it shorter rather than guess."
- "short_definition must be quotable from the source. clinical_context may synthesize but must be grounded in the source."

### Step 3 — Manual review (the actual quality gate)

The curator opens `review_worksheet.csv` in a spreadsheet. For each entry:

- **accept** (1/0): take unchanged
- **edit**: corrected text inline
- **reject** (1/0): drop completely

Workflow is explicitly **not** "carefully read all 300 entries". That is
unrealistic. Instead:

1. Sort by `category` and review one category per session
2. Per category: 3–5 random spot-checks against the cited source, others
   skimmed for plausibility
3. If a hit (hallucination, wrong number, wrong source) → review the whole
   category in detail
4. On clean spot-check → bulk-accept

Realistic time: ~3–6 hours for 300 entries, given good generation.

### Step 4 — Merge

`scripts/curation/review_worksheet_to_yaml.py`:

1. Reads reviewed CSV
2. Drops `reject=1` rows
3. Applies edits
4. Sets `review_status="reviewed"` and `last_reviewed=today` for `accept=1`
5. Merges into `app/knowledge/oncology/lexicon.yaml`
6. Validates the entire merged lexicon against the schema
7. Generates a diff report (what's new, what changed)

### Quality gates (enforced before merge)

| Gate | Implemented in | Prevents |
|---|---|---|
| Schema validation | Pydantic in loader.py | Structural errors, missing required fields |
| Source ≥1 required | Pydantic field validator | Sourceless hallucinations |
| Source domain allowlist | Pydantic field validator | Fabricated URLs / non-authoritative sources |
| Alias uniqueness | test_lexicon_quality.py | Term-matching conflicts |
| related_terms referential integrity | test_lexicon_quality.py | Dead links between entries |
| Manual review marker | review_status field | Unreviewed entries marked in UI/log |
| Diff review pre-commit | git diff visual | Final human eyeball check |

### Hallucination risk mitigation

| Risk | Mitigation |
|---|---|
| LLM fabricates source URL | Domain allowlist (Pydantic validator) rejects unknown domains |
| LLM cites real source but wrong content | Spot-check workflow in review |
| LLM mixes outdated and current definitions | `last_reviewed` makes aging visible; v2 may add re-review loop |
| Editor misses subtle hallucination | `review_status="reviewed"` ≠ `expert-approved`. Three-stage lifecycle allows later expert pass without blocking v1 delivery |
| LLM generation is non-deterministic | Generation runs rarely (not in CI), output is committed and not regenerated. Determinism via repo versioning. |

## 7. Enrichment Logic

### When does enrichment run?

Exactly once per tool call, at the end of orchestration, before the envelope:

```
MCP Tool Call → Adapter Fan-out → Pydantic models
                                          │
                                          ▼
                              orchestrator.enrich(response)
                                          │
                                          ▼
                            Models now have knowledge_annotations
                                          │
                                          ▼
                              decision.pick_recipe() (WS1)
                                          │
                                          ▼
                              envelope_to_llm_text() (WS1 guarantor)
```

Enrichment is **side-effect-free**: it produces a new response instance with
additional annotation fields, never mutating the original. Idempotent.

### Annotation model

```python
class Annotation(BaseModel):
    field_path: str           # e.g. "trial.phase", "trial.endpoints[0].name"
    matched_term: str         # the literal matched string, e.g. "Phase 3"
    lexicon_id: str           # e.g. "trial-phase-3"
    short_definition: str     # copied from LexiconEntry for direct frontend access
    clinical_context: str
    review_status: str        # "llm-generated" | "reviewed" | "expert-approved"
```

`field_path` lets recipes anchor the annotation to the exact field where the
match occurred (tooltip on the phase label, not generic at the bottom).

### Matching strategy

`app/services/enrichment.py`:

```python
def enrich(response: BaseToolResponse, lexicon: Lexicon) -> BaseToolResponse:
    annotations: list[Annotation] = []
    for record in iter_records(response):
        for field_path, value in iter_string_fields(record):
            for term, entry in lexicon.term_index.items():
                if _matches(value, term, entry.aliases):
                    annotations.append(Annotation(
                        field_path=field_path,
                        matched_term=term,
                        lexicon_id=entry.id,
                        short_definition=entry.short_definition,
                        clinical_context=entry.clinical_context,
                        review_status=entry.review_status,
                    ))
    return response.copy(update={"knowledge_annotations": annotations})
```

**Match rules** (in `_matches()`):

1. Case-insensitive exact match on canonical term or alias
2. Word boundary (so "RECIST" does not match in "prerecisted")
3. No fuzzy matching, no substring matching, no NLP — deliberately simple
4. First match wins per `field_path` → no duplicate annotations on same field
5. Maximum N annotations per response (default: 50) → prevents noise on large responses

The simplicity is intentional: the lexicon is small (~300) and curated
(canonical terms + explicit aliases). Fuzzy matching would introduce false
positives that the controlled vocabulary does not need.

### Performance

- Lexicon held in memory (loaded at startup)
- Term index is a dict, O(1) per lookup
- Per response: `iter_records × iter_fields × len(term_index)` ≈ 12 trials × 20 fields × 300 terms = ~72k comparisons → microseconds in Python
- No latency concerns. If a hotspot ever appears, Aho-Corasick is a drop-in replacement.

### Visual surfacing (recipe integration)

Recipes consume `knowledge_annotations` **opt-in**. No recipe is required to
show them.

| Pattern | Recipe | Display |
|---|---|---|
| Tooltip on field | gantt, single-entity-card | Hover on "Phase 3" shows short_definition |
| Footnote block at end | html_cards, info-card | List of used terms as a glossary |
| Inline sublabel | concept-card | Definition directly under the term |

Each recipe may filter further (e.g., only show `review_status != "llm-generated"`,
or only the top 5).

### Data flow examples

**Example 1**: query "List Phase 3 Pembrolizumab trials in NSCLC"

```
1. search_trials_with_publications → 12 trials
2. enrich() walks records → matches:
   - "Phase 3" → trial-phase-3 (deduplicated to 1)
   - "NSCLC" → tumor-type-nsclc
   - "Pembrolizumab" → drug-pembrolizumab (drug-class match)
   - "Overall Survival" in endpoints → endpoint-os
3. Response has 4 unique annotations
4. Gantt recipe shows tooltips on phase label and endpoint column
5. LLM receives the enriched Pydantic JSON + the rendered artifact
```

**Example 2**: query "What is RECIST?"

```
1. search_pubmed → short definition text
2. enrich() matches "RECIST" → response-criterion-recist-1-1
3. WS1 fallback: concept-card recipe is chosen
4. concept-card consumes the annotation → renders clinical_context as sublabel
5. LLM pastes the artifact, which already contains a rich definition
```

### Out of scope (enrichment)

- No reasoning over annotations (the LLM gets them as data, the server does not interpret)
- No cross-annotation logic (e.g., combining "Phase 3 + Pembrolizumab + NSCLC" into a story)
- No language translation
- No annotation highlighting in the LLM text output (artifacts only)

## 8. Testing Strategy

### Schema & loader tests — `tests/knowledge/test_lexicon_load.py`

```python
def test_lexicon_loads_clean():
    lex = load_lexicon()
    assert len(lex.entries) >= 250  # safety floor

def test_all_entries_have_sources():
    lex = load_lexicon()
    for entry in lex.entries:
        assert len(entry.sources) >= 1, f"{entry.id} has no sources"

def test_source_domains_are_authoritative():
    ALLOWED = {"ncit.nci.nih.gov", "fda.gov", "ema.europa.eu",
               "recist.eortc.org", "pubmed.ncbi.nlm.nih.gov", "ctep.cancer.gov"}
    lex = load_lexicon()
    for entry in lex.entries:
        for source in entry.sources:
            if source.url:
                domain = urlparse(source.url).netloc
                assert domain in ALLOWED, f"{entry.id}: bad source {domain}"
```

### Lexicon quality tests — `tests/knowledge/test_lexicon_quality.py`

| Test | Checks |
|---|---|
| `test_no_duplicate_ids` | No duplicate ids |
| `test_no_duplicate_aliases` | No two entries share an alias |
| `test_related_terms_resolve` | All `related_terms` reference existing ids |
| `test_categories_balanced` | No empty category; each has ≥3 entries |

### Enrichment logic tests — `tests/services/test_enrichment.py`

```python
def test_enrich_matches_canonical_term():
    response = make_trial_response(phase="Phase 3")
    enriched = enrich(response, test_lexicon)
    assert any(a.lexicon_id == "trial-phase-3"
               for a in enriched.knowledge_annotations)

def test_enrich_matches_alias():
    response = make_trial_response(endpoint="OS")
    enriched = enrich(response, test_lexicon)
    assert any(a.matched_term == "OS" and a.lexicon_id == "endpoint-os"
               for a in enriched.knowledge_annotations)

def test_enrich_word_boundary():
    response = make_trial_response(notes="prerecisted criteria")
    enriched = enrich(response, test_lexicon)
    assert not any(a.lexicon_id == "response-criterion-recist-1-1"
                   for a in enriched.knowledge_annotations)

def test_enrich_idempotent():
    response = make_trial_response(phase="Phase 3")
    once = enrich(response, test_lexicon)
    twice = enrich(once, test_lexicon)
    assert once.knowledge_annotations == twice.knowledge_annotations

def test_enrich_max_annotations_cap():
    response = make_huge_response(num_records=50)
    enriched = enrich(response, test_lexicon)
    assert len(enriched.knowledge_annotations) <= 50

def test_enrich_does_not_mutate_original():
    response = make_trial_response(phase="Phase 3")
    original_dict = response.model_dump()
    _ = enrich(response, test_lexicon)
    assert response.model_dump() == original_dict
```

- `generate_lexicon.py` produces schema-conforming YAML from a 3-term mini seed
- `review_worksheet_to_yaml.py` merges correctly: rejected entries gone, edits applied, `review_status` set
- Validation runs at the end of merge and fails on broken schema

### Integration test — `tests/test_orchestration_with_enrichment.py`

End-to-end: a mock tool call → orchestrator → enrichment → envelope → final

### Out of test scope

- Tooltip visual correctness in recipes (snapshot suffices)
- LLM behavior with annotations
- Performance benchmarks
- Content correctness of lexicon entries (this is the manual review's job, not a test's)

## 9. Phasing

| Phase | Content | Verification |
|---|---|---|
| 0 | Schema + loader (empty lexicon) + tests | Schema tests green; lexicon.yaml empty but loads |
| 1 | Manually maintain `seed_topics.yaml` (~300 terms) | YAML loads cleanly |
| 2 | Write `generate_lexicon.py` + mini-seed test | 3 test terms generated + validated |

| 4 | **Manual review** (off-cycle, by user) | reviewed CSV with accept/edit/reject markings |
| 5 | `review_worksheet_to_yaml.py` → final lexicon.yaml | Quality tests green, domain allowlist green |
| 6 | `app/services/enrichment.py` + tests | All enrichment tests green |
| 7 | Orchestrator wiring (lexicon singleton + enrich call) | Integration test green |
| 8 | Recipe integration (tooltip in gantt + footnote in info-card) | Snapshot tests, manual visual spot-check |

Phases 0–2 and 6–8 are pure development. Phases 3–5 are curation work
(LLM call + manual review). Curation is the only bottleneck — it needs real
focus but is bounded to a single sitting block.

## 10. Risks

| Risk | Severity | Mitigation |
|---|---|---|
| Manual review skipped or rushed | High | `review_status` lifecycle makes it transparent. Frontend / coverage_log marks `llm-generated` annotations as "unverified". Can be reviewed later. |
| LLM fabricates plausible but wrong clinical_context | Medium | Source allowlist + spot-check workflow + visible review_status. Accepted residual risk in v1. |
| Term matcher too greedy or too sparse | Low | Word boundary + first-match-wins. Tests cover the most common edge cases. v2 may add weighting. |
| Lexicon goes stale | Medium long-term | `last_reviewed` makes aging visible. v2: re-review workflow every 12 months. v1 explicitly ignores this. |
| Annotation noise floods the output | Low | Cap of 50 per response, deduplicated by `lexicon_id`. Recipes filter further. |
| NCI Thesaurus license | Low | NCIt is public domain (US Government Work). FDA/EMA are too. RECIST publications can be cited. No license concerns expected. |
| Drift between WS1 (coverage) and WS2 (enrichment) | Low | Both write into the same Pydantic model. Integration test catches conflicts. |

## 11. Success Criteria

1. Lexicon loads with ≥250 entries, all schema-valid.
2. All quality tests green: no duplicate ids, no alias collisions, all source domains in allowlist, all related_terms resolve.
3. All enrichment tests green: canonical match, alias match, word boundary, idempotence, cap, no mutation.
4. End-to-end integration test passes with both WS1 (artifact present) and WS2 (annotations attached).
5. Manual spot-check: send 5 oncology queries, see annotations rendered in at least one recipe (tooltip or footnote).
6. **Monitored metric** (not a hard gate): ≥80% of generated entries pass manual review on first pass — used to judge whether the generation prompt needs tuning, not to block delivery.
